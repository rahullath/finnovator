# Day-of Workflow

## Hour 0–1: Orient (don't skip this)
1. Find out which challenge you're on
2. Open the relevant spec folder
3. Everyone reads the spec — yes all of you, 15 minutes
4. Finance people: read the methodology section and flag anything they can't defend
5. Tech people: check API access works (run the test scripts in `/shared/api-tests/`)
6. Agree on ONE idea from the spec — don't try to build everything

## Hour 1–2: Scaffold fast
- Tech lead initialises the repo from the template in `/shared/boilerplate/`
- Supabase project created, `.env` populated
- One working API call to your primary data source
- Don't design the UI yet — get data flowing first

## Hour 2–6 (Saturday): Build core
- Data pipeline → scoring logic → basic UI
- Finance people: write the methodology doc (1 page, for the pitch)
- Check in every 90 minutes: "does this still make sense?"

## Hour 6–9 (Saturday): Demo path
- Identify ONE user journey that shows the core value
- Make that journey work end-to-end, even if everything else is rough
- Don't add features — polish the demo path

## Sunday: Pitch prep
- Working demo (not slides — a live product)
- 3-minute pitch structure: problem → why existing solutions fail → your approach → demo → scalability
- Prepare for: "why is this different from X?" and "what's the business model?"

---

## What judges reward (from Birmingham winner analysis)
1. Working prototype > polished slides
2. Specific, real pain point — not "ESG is broken" but "here is the exact gap"
3. User-first framing — who uses this and why does it help them
4. Investment readiness — hint at how this scales

## What kills teams
- Spending hour 1 arguing about the idea
- Building features instead of the demo path
- Tech explaining the product to judges (finance people should pitch)
- Static mockups presented as "prototypes"
